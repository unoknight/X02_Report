## 2. Tổng quan nghiệp vụ màn hình

Màn hình dùng để tạo và quản lý bản ghi ngành đào tạo mà một trường/cơ sở đào tạo được phép đào tạo. Một bản ghi chính được định danh nghiệp vụ bởi tổ hợp: trường, ngành đào tạo, trình độ, hình thức đào tạo. Khi tạo mới, người dùng chọn ngành, nhóm ngành, trình độ, hình thức, năm bắt đầu, trạng thái đào tạo và có thể liên kết văn bản/quyết định mở ngành.

Màn hình có workflow phê duyệt theo metadata và backend: bản ghi tạo ở `Draft`, người dùng role `Officer` gửi duyệt sang `Submitted`, role `Approver` phê duyệt sang `Approved` hoặc từ chối sang `Rejected`. Rule maker-checker chi tiết chưa đủ căn cứ để kết luận, cần xác nhận thêm.

## 6. Màn hình Create / Detail / Edit

- `Thông tin cơ bản`: ngành đào tạo, nhóm ngành đào tạo, trình độ đào tạo, hình thức đào tạo, năm bắt đầu đào tạo, trạng thái đào tạo, tên cơ sở đào tạo.

### 9.2 Rule phân quyền theo người tạo

Cần xác nhận rule maker-checker: người tạo có được tự approve hay không. Khuyến nghị không cho cùng một user vừa submit vừa approve để giảm rủi ro kiểm soát nội bộ.

### 9.3 Rule phân quyền theo trạng thái bản ghi

| Status | Role | Action được phép | Action không được phép | Ghi chú |
|---|---|---|---|---|
| Draft | Officer | Edit, Delete, Submit | Approve, Reject | Delete cần xác nhận vì metadata mâu thuẫn. |
| Submitted | Approver | Approve, Reject, View | Edit, Delete | Officer chỉ View/Cancel nếu nghiệp vụ cho phép. |
| Approved | All authorized | View, Export | Edit, Delete, Submit | Metadata ẩn Edit/Delete khi Approved. |
| Rejected | Officer | Edit, Submit, Delete | Approve | |
| Locked | Authorized | View | Edit/Delete/Submit/Approve/Reject | Cần xác nhận rule unlock. |

### 9.4 Rule phân quyền theo phòng ban/đơn vị

Backend có logic lấy trường theo `MA_DON_VI` user khi create. Search hiện comment bỏ scope cứng theo trường; cần xác nhận có bắt buộc giới hạn dữ liệu theo `MA_DON_VI`/trường hay không. Khuyến nghị backend luôn kiểm tra scope đơn vị, frontend chỉ dùng để ẩn/hiện.

## 15. Acceptance Criteria

### AC-01: Xem danh sách bản ghi
Given user có quyền xem  
When user mở màn List ngành đào tạo  
Then hệ thống hiển thị danh sách bản ghi không bị xóa mềm và đúng phạm vi quyền.

### AC-02: Tìm kiếm dữ liệu
Given user nhập điều kiện tìm kiếm hợp lệ  
When user thực hiện search  
Then hệ thống trả dữ liệu khớp điều kiện và không lỗi do field cấu hình không tồn tại.

### AC-03: Lọc dữ liệu
Given user chọn filter trạng thái/trường/trình độ/hình thức  
When user áp dụng filter  
Then danh sách chỉ hiển thị bản ghi khớp filter.

### AC-04: Sắp xếp dữ liệu
Given user click sort ở cột được hỗ trợ  
When hệ thống tải lại danh sách  
Then dữ liệu được sắp xếp đúng ASC/DESC.

### AC-05: Phân trang
Given danh sách có nhiều bản ghi  
When user chuyển trang  
Then hệ thống hiển thị đúng page size và tổng số bản ghi.

### AC-06: Thêm mới bản ghi hợp lệ
Given user có quyền tạo và trường của user đã Approved  
When user nhập đầy đủ thông tin hợp lệ và bấm Lưu  
Then hệ thống tạo bản ghi `Draft` trong `TRUONG_NGANH_DAO_TAO`.

### AC-07: Validate field bắt buộc
Given user bỏ trống field bắt buộc  
When user bấm Lưu hoặc Submit  
Then hệ thống hiển thị message lỗi tiếng Việt tại field tương ứng.

### AC-08: Validate format dữ liệu
Given user nhập năm bắt đầu không hợp lệ  
When user bấm Lưu  
Then hệ thống báo năm bắt đầu phải trong khoảng hợp lệ.

### AC-09: Validate trùng dữ liệu
Given đã tồn tại bản ghi cùng trường/ngành/trình độ/hình thức chưa bị xóa  
When user tạo bản ghi trùng  
Then hệ thống từ chối lưu và báo dữ liệu đã tồn tại.

### AC-10: Xem chi tiết bản ghi
Given user có quyền xem bản ghi  
When user mở Detail  
Then hệ thống hiển thị dữ liệu chính, label lookup, trạng thái và audit nếu có quyền.

### AC-11: Chỉnh sửa bản ghi
Given bản ghi ở Draft/Rejected và user có quyền sửa  
When user cập nhật dữ liệu hợp lệ  
Then hệ thống lưu thay đổi, tăng version và ghi audit.

### AC-12: Submit phê duyệt
Given bản ghi ở Draft/Rejected và đủ dữ liệu bắt buộc  
When Officer bấm Submit  
Then bản ghi chuyển sang Submitted và ghi log SUBMIT.

### AC-13: Approve bản ghi
Given bản ghi ở Submitted  
When Approver bấm Approve  
Then bản ghi chuyển sang Approved và ghi người/ngày duyệt.

### AC-14: Reject bản ghi
Given bản ghi ở Submitted  
When Approver nhập lý do và bấm Reject  
Then bản ghi chuyển sang Rejected và lưu lý do từ chối.

### AC-15: Kiểm tra phân quyền
Given user không có quyền action  
When user gọi API action trực tiếp  
Then backend trả 403 và không cập nhật dữ liệu.

### AC-16: Kiểm tra lỗi không có quyền
Given user không thuộc phạm vi đơn vị/trường  
When user truy cập bản ghi ngoài phạm vi  
Then hệ thống không trả dữ liệu hoặc trả 403.

### AC-17: Xem lịch sử phê duyệt
Given user có quyền xem audit  
When user mở tab Lịch sử chỉnh sửa  
Then hệ thống hiển thị các action Create/Update/Submit/Approve/Reject.

### AC-18: Ghi audit log
Given có thao tác thay đổi dữ liệu hoặc workflow  
When thao tác thành công  
Then hệ thống ghi bản ghi tương ứng vào `LICH_SU_THAY_DOI`.
